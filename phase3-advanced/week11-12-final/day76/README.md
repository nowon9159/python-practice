# Day 76: 시스템 설계 심화 — 실시간 채팅 + WebSocket

## 🎯 오늘의 목표

실시간 채팅 시스템의 요구사항을 정의하고, WebSocket(또는 long polling)과 메시지 브로커를 활용한 아키텍처 다이어그램·메시지 흐름을 설계 문서로 정리한다.

---

## ✅ 오늘 할 일 (순서대로)

### 1. 요구사항 정의

- **할 일**: `phase3-advanced/week11-12-final/day76/` 안에 `design.md`(또는 `chat_design.md`)를 만든다. **요구사항**을 적는다. (1:1 채팅, 그룹 채팅, 히스토리 조회, 읽음 표시, 다중 디바이스, 오프라인 메시지 등 중 3개 이상.)
- **완료 조건**: 문서에 요구사항 3개 이상이 명시되어 있다.

### 2. 아키텍처 다이어그램 + 메시지 흐름

- **할 일**: 같은 문서에 **아키텍처 다이어그램**을 추가한다. 클라이언트–서버, WebSocket(또는 long polling) 연결, 메시지 브로커(Redis Pub/Sub 등)로 다중 서버 확장하는 흐름을 그린다. **실시간 전달·확장 방식**을 2~3문장으로 설명한다.
- **완료 조건**: 다이어그램과 설명으로 "메시지가 어떻게 전달되는지, 서버를 늘리면 어떻게 되는지"가 드러난다.

### 3. (선택) WebSocket 에코 서버·Anki

- **할 일**: (선택) `day76/echo_server.py` 등으로 간단한 WebSocket 에코 서버를 구현해 연결·메시지 송수신을 확인. Anki에 "채팅 시스템 설계"(실시간 전달, 확장 방식) 카드 추가.
- **완료 조건**: 설계 문서가 완성되었고, 실시간 전달·확장 방식을 설명할 수 있으며, (선택) 에코 서버를 실행해 봤다. Anki 카드가 추가되어 있다.

## 📌 참고 (복습할 개념)

- WebSocket: 양방향, 지속 연결
- 메시지 브로커(Redis Pub/Sub 등): 다중 서버에서 메시지 공유
- 오프라인 메시지, 읽음 표시 등 부가 요구사항
